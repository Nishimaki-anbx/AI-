# AI 開発プレイブック

このガイドでは、AI 支援開発を有用でレビューしやすく、安全な形で練習する方法を説明します。

## 目的

- AI を未確認の生成器としてではなく、協力者として使う
- 判断の経緯を GitHub 上で見える状態にする
- 小さく変更し、すばやくレビューする

## 推奨ワークフロー

### 1. Issue で目的を絞る

弱い例:

- 何か機能を追加する

よりよい例:

- README にセットアップ手順を追加する
- CI で Markdown と YAML を検証する
- Pull Request テンプレートに AI レビュー欄を追加する

### 2. AI への依頼は具体的にする

よい依頼には、通常次の内容が含まれます。

- 背景
- 完了条件
- 変更してよいファイルや領域
- 変更してはいけないファイルや領域
- 注意してレビューしてほしい点

### 3. 作業を小さな単位に分ける

典型的な単位:

- ドキュメント更新
- Issue と PR のテンプレート
- GitHub Actions の設定
- サンプル実装

### 4. 最後は人間がレビューする

特に次の点に注意します。

- 要件の誤解
- 名前付けと責務の境界
- セキュリティと公開範囲のリスク
- 後から学習材料として読める状態になっているか

## PR に記録すること

- AI への依頼内容の要約
- AI の提案から選んだ方針
- 採用しなかった案とその理由
- 人間が実施した手動チェック

## 避けること

- 非常に大きな Pull Request
- Issue のない追跡不能な作業
- レビューなしで AI の出力をマージすること
- 重要な判断をチャットだけに残し、GitHub に残さないこと
