# コントリビューションガイド

このリポジトリは、AI 支援開発を安全かつ繰り返し練習できるように設計されています。

## 基本ルール

- 1つの Issue は1つの成果に集中させる
- 1つの Pull Request は1つの論理的な変更に集中させる
- AI による支援と人間による検証は分けて記録する
- 目的があいまいな場合は、まずドキュメントを更新して範囲を絞る

## 標準ワークフロー

1. Issue を作成する、または選択する
2. `main` からブランチを作成する
3. ローカルで実装、ドキュメント更新、検証を行う
4. Pull Request を作成する
5. CI の結果を確認する
6. レビュー後、squash merge で `main` にマージする

## ブランチ名

推奨パターン:

- `feat/123-short-title`
- `fix/123-short-title`
- `docs/123-short-title`
- `chore/123-short-title`

まだ Issue がない場合でも、短く内容が分かるタイトルを含めます。

## コミットメッセージ

推奨スタイル:

- `feat: add initial issue templates`
- `docs: document AI development workflow`
- `chore: add repository hygiene workflow`

コミットは後から読んでも理解できる内容にします。共有ブランチには、長く続く WIP の履歴を残さないようにします。

## Pull Request

各 Pull Request では、次の内容を説明します。

- 何を変更したか
- なぜ変更したか
- AI をどのように使ったか
- 人間が何を検証したか
- 未対応または後回しにしたことは何か

`.github/PULL_REQUEST_TEMPLATE.md` を基本形として使います。

## レビューの優先事項

レビューでは、次の点に注目します。

- Issue の目的が実際に満たされているか
- スコープが適切か
- 判断理由が明確に記録されているか
- AI が生成した重要な変更を人間が確認しているか
- 後から学習材料として役立つ履歴になっているか

## AI 利用ルール

- 必要に応じて、重要なプロンプトや指示を PR または Issue に要約する
- AI の出力をそのまま受け入れず、意図、影響、トレードオフを確認する
- 大きな変更は小さな段階に分ける
- AI ツールにコンテキストを送る前に、シークレット、認証情報、個人情報を確認する

## マージ方針

- `main` への直接 push は避ける
- GitHub 上でレビューと CI が完了してからマージする
- 履歴を読みやすく保つため、`Squash merge` を優先する
